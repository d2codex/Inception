## Description

**Inception** is a project from the 42 curriculum focused on building a small, secure, and modular infrastructure using **Docker** and **Docker Compose**.

The goal of this project is to learn how to create and manage a multi-container environment by building custom Docker images, configuring isolated services, and handling communication between containers.

The infrastructure is composed of three main services:

- **NGINX**
  - Acts as the only entry point to the infrastructure.
  - Handles HTTPS connections using **TLSv1.2/TLSv1.3**.
  - Redirects requests to the WordPress service.

- **WordPress + PHP-FPM**
  - Provides the application layer.
  - Runs WordPress with PHP-FPM only, without an integrated web server.

- **MariaDB**
  - Provides the database layer.
  - Stores WordPress data and user information.

### Infrastructure

The services communicate through a dedicated **Docker network** to ensure that
data remains available even when containers are recreated.

Persistent data is managed using **Docker named volumes**:
- MariaDB volume: stores the database files.
- WordPress volume: stores website files.

## Design Choices

- Built the infrastructure using **Docker Compose** to orchestrate multiple independent services.
- Created **custom Docker images** for NGINX, WordPress, and MariaDB as required by the subject.
- Separated responsibilities by running **one service per container**.
- Used a **dedicated Docker network** for communication between services while exposing only NGINX externally.
- Used **Docker named volumes** to persist WordPress files and MariaDB data.
- Used **Docker secrets** for sensitive credentials, with each secret stored in a separate file.
  Secrets are automatically generated by the Makefile during setup to avoid storing passwords in the repository.
- Configured **NGINX as the only entry point** using HTTPS with TLSv1.2/TLSv1.3.

### Virtual Machines vs Docker

| Virtual Machines                              | Docker                                            |
| --------------------------------------------- | ------------------------------------------------- |
| Runs a complete OS with its own kernel.       | Runs isolated containers sharing the host kernel. |
| Higher resource usage.                        | Lightweight and efficient.                        |
| Slower startup time.                          | Fast startup time.                                |
| Strong OS-level isolation.                    | Application-level isolation.                      |
| Best for running different operating systems. | Best for deploying and managing services.         |

### Secrets vs Environment Variables

| | Environment Variables | Docker Secrets |
|---|---|---|
| Purpose | General configuration | Sensitive data (passwords, keys) |
| Storage | `.env` file | Mounted as files in `/run/secrets` |
| Security | Visible to container processes | Not exposed as environment variables |
| Used in this project | Database name, usernames, domain | Database and WordPress passwords |

### Docker Network vs Host Network

| | Docker Network | Host Network |
|---|---|---|
| Isolation | Containers communicate on a private network | Containers share the host's network |
| Port Exposure | Only published ports are accessible | Services are exposed directly on the host |
| Security | Better isolation between services | Reduced isolation |
| Used in this project | Dedicated bridge network connecting the containers | Not used (forbidden by the subject) |

### Docker Volumes vs Bind Mounts

| | Docker Volumes | Bind Mounts |
|---|---|---|
| Managed By | Docker | Host filesystem |
| Portability | Easy to move and back up | Tied to a specific host path |
| Typical Use | Persistent application data | Source code and configuration files |
| Used in this project | WordPress files and MariaDB data | Configuration files only (Dockerfiles, NGINX config, etc.) |

## Instructions

### Prerequisites

- Docker
- Docker Compose

### Setup

1. Clone the repository.
2. Configure the following variables in `srcs/.env`:

```env
DOMAIN_NAME=<login>.42.fr

MYSQL_DATABASE=<database_name>
MYSQL_USER=<database_user>

WP_TITLE=<website_title>
WP_ADMIN_USER=<administrator_username>
WP_ADMIN_EMAIL=<administrator_email>

WP_USER=<wordpress_username>
WP_USER_EMAIL=<wordpress_user_email>
```

4. Ensure `<login>.42.fr` resolves to your local machine.

### How to resolve

To access the website locally, map your domain name to the machine running Docker.

**Local workstation**

Add the following entry to your hosts file:

```text
127.0.0.1    <login>.42.fr
```

**42 school machines**

If Docker is running on a remote/headless machine, map `<login>.42.fr` to the remote machine's IP
address or use SSH/X11 forwarding, depending on your setup.

### Build and Run

```bash
make
```

### Stop the Project

```bash
make down
```

### Remove Containers, Images and Volumes

```bash
make clean
```

or, to completely reset the project:

```bash
make fclean
```
